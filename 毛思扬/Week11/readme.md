# 作业2:

对多模态RAG的项目，如果用户使用【文本】提问 vs 【文本 + 图】提问，你会怎么处理？有什么区别？

## 处理方式差异

### 文本-only提问处理
- 只使用question_text字段进行处理
- 仅对文本查询进行语义编码和向量检索
- 在`multimodal_qa`接口中，只处理纯文本语义理解

### 文本+图像提问处理
- 同时使用question_text和question_image字段
- 需要对文本和图像进行多模态融合处理
- 使用CLIP等多模态模型将文本和图像映射到统一语义空间

## 主要区别

1. **特征提取方式不同**
   - 纯文本：仅使用NLP模型提取文本特征
   - 文本+图像：需要同时提取文本和视觉特征，并进行融合

2. **检索策略不同**
   - 纯文本查询：在文本向量空间中检索
   - 多模态查询：在统一的多模态向量空间中检索

3. **计算复杂度不同**
   - 文本-only相对简单，计算成本较低
   - 多模态查询涉及更多计算，需要处理图像编码

4. **语义理解深度不同**
   - 纯文本依赖语言模型理解
   - 多模态能结合视觉信息提供更丰富的语义理解


# 作业3:

对多模态RAG的项目，milvus如果要存储文本、图、表，数据结构如何设计？

## 统一向量存储设计

### 1. 公共字段设计
- `id`: 主键ID，唯一标识每个数据项
- `embeddingfile`: 向量字段，存储统一语义空间的向量表示
- `content_type`: 内容类型标识（text/image/table）
- `source_id`: 原始数据源ID，关联到具体存储位置
- `metadata`: JSON字段，存储各类型特有的元数据

### 2. 文本数据结构
- `text_content`: 存储原始文本内容
- `metadata`: 
  - `title`: 文档标题
  - `language`: 语言类型
  - `word_count`: 字数统计
  - `tags`: 标签信息

### 3. 图像数据结构
- `image_path`: 图像文件存储路径
- `metadata`:
  - `width`: 图像宽度
  - `height`: 图像高度
  - `format`: 图像格式
  - `caption`: 图像描述（如果有）

### 4. 表格数据结构
- `table_content`: 表格数据的结构化存储（JSON格式）
- `metadata`:
  - `rows`: 行数
  - `columns`: 列数
  - `headers`: 表头信息
  - `table_caption`: 表格标题
